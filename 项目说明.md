# 项目说明与局限性声明

## 项目性质声明

**本项目是一套工作流设计文档，而非可直接执行的代码项目。**

项目包含的内容：
- 8个 SKILL.md 描述文件（Trae Agent 技能定义）
- 8个 hermes_skill.json 文件（Hermes Agent 兼容格式）
- 8个 openclaw_skill.json 文件（OpenCLAW 兼容格式）
- 配套文档（使用手册、技术配置、集成指南等）

项目**不包含**的内容：
- ❌ 实际爬虫代码
- ❌ LLM 调用代码
- ❌ 质量评分代码
- ❌ 任何可执行代码

## 数据源限制说明

本项目提到的数据抓取功能（起点、番茄、七猫等平台）仅为**设计理念和策略指导**：

1. **实际可行性有限**：这些平台都有严格的反爬机制/验证码，实际可获取的数据非常有限
2. **合规要求**：任何数据收集都必须遵守平台 robots.txt 和服务条款
3. **替代方案**：建议通过以下合法方式获取市场数据：
   - 平台官方公开排行榜（手动查看）
   - 第三方行业报告（艾瑞、QuestMobile等）
   - 社交媒体趋势分析
   - 作者社区讨论

## API 批量生成策略

当使用 LLM API 批量生成大量章节时，需要关注以下实际问题：

### 1. 长输出稳定性
- **问题**：API 在生成长文本时容易出现内容质量下降
- **解决方案**：
  - 分章节生成，每章独立调用 API
  - 每章 2000-3000 字为宜，不要一次性生成多章
  - 使用 `max_new_tokens` 参数控制输出长度

### 2. 主角名漂移问题
- **问题**：随着章节增多，主角名字可能漂移或变化
- **解决方案**：
  - 维护人物卡片（character card），每次请求都带上人物信息
  - 使用系统提示词（system prompt）固定主角姓名和特征
  - 每生成 10 章后检查人物一致性

### 3. API 速率限制
- **问题**：大多数 API 有速率限制（如每分钟 60 次请求）
- **解决方案**：
  - 实施请求间隔（建议 2-5 秒）
  - 使用指数退避重试策略
  - 批量生成时控制并发数（建议不超过 3 个并发）

### 4. 设定一致性维护
- **问题**：AI 在长文本生成中容易遗忘前期设定
- **解决方案**：
  - 维护设定文档（setting doc），包含世界观、力量体系等
  - 每次生成章节时，在提示词中包含关键设定摘要
  - 定期（每 30 章）重新生成设定摘要

## 关于"AI去痕化"

**说明**：本项目最初设计的"AI痕迹检测"概念实际上并非读者真正关心的点。

实际情况：
- 读者只在乎内容好不好看，不在乎是 AI 写的还是人工写的
- "AI痕迹"更多是开发者角度的质量问题

因此，建议将"AI痕迹检测"改为**"可读性评分"**，关注：
- 内容是否流畅
- 情节是否吸引人
- 人物是否立体
- 节奏是否合理

## 项目使用建议

### 适合的用途
1. 作为网文创作的**设计指南**和**灵感参考**
2. 理解网文创作流程的**方法论框架**
3. 配合 LLM API 进行**辅助创作**

### 不适合的用途
1. 作为全自动生成工具（实际效果不可控）
2. 作为数据抓取工具（实际可行性有限）
3. 期望一键生成完整小说（质量无法保证）

## 推荐的实际使用流程

1. **市场调研**：手动查看各平台排行榜和行业报告
2. **选题分析**：参考选题分析框架，结合个人兴趣确定方向
3. **设计方案**：使用设计规划技能辅助构思人物和世界观
4. **大纲规划**：参考大纲框架，规划章节结构
5. **辅助创作**：使用 LLM 逐章生成内容草稿
6. **人工润色**：**必须进行人工编辑和润色**
7. **发布前审核**：检查设定一致性、情节连贯性

**关键**：AI 只是辅助工具，高质量内容需要人工深度参与。
